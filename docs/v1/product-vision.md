# v1 产品愿景

## 一句话定位

`llm-tracelab v1` 是一个本地优先、可自托管的 LLM 行为观测、审计、调试与回放工作台。

它像 AI 应用的 flight recorder：真实记录每一次 LLM 调用，保留原始证据，理解协议内容，识别风险行为，并支持复现和长期分析。

## 为什么需要 v1

当前 AI 应用越来越黑盒。出了问题时，开发者往往只能看到：

- 一段 prompt。
- 一个模型最终回答。
- 一些 SDK 日志。
- 一个 provider dashboard 的粗粒度统计。

这些信息不足以回答真正的排查问题：

- 应用到底传给模型了什么？
- 哪个系统指令、工具声明、上下文片段影响了结果？
- 模型是否调用了工具？调用参数是什么？
- 工具结果是什么？是否包含错误或敏感信息？
- 模型是否试图执行危险命令或修改关键文件？
- 请求被哪个上游处理？为什么路由到了它？
- TTFT、生成速率、缓存率、token 消耗是否异常？
- 同一任务在不同模型、不同 provider 上行为有什么差异？

v0.x 已经解决了 raw 录制、回放、基础查看、多上游路由等问题。v1 要解决的是：从 raw 流量中还原 AI 应用行为。

## 目标用户

### 个人高级 AI 使用者

用于：

- 调试 Codex、Claude Code、Gemini CLI、OpenAI SDK 等工具行为。
- 对比不同模型的真实能力和工具调用差异。
- 从长期使用记录中总结更高效的 AI 工作方法。
- 复用真实 LLM 请求作为单元测试 fixture。

### AI 应用开发者

用于：

- 本地开发时录制/回放 LLM 请求。
- 定位 prompt、tool call、streaming、provider error、缓存命中等问题。
- 在不消耗 token 的情况下复现问题。
- 建立回归测试数据集。

### 企业 AI 平台/安全团队

用于：

- 自托管 LLM 代理网关。
- 审计危险工具调用和敏感信息泄露。
- 观察多上游可用性和性能。
- 为 AI 应用提供本地/内网可控的行为证据链。

## 核心问题

v1 应该持续回答八类问题：

1. 请求传输了什么：system/developer/user/tool/content/file/image/audio。
2. 模型返回了什么：text/reasoning/tool call/tool result/refusal/safety block。
3. 工具调用了什么：工具名、参数、结果、错误、执行归属。
4. 是否有风险：危险命令、敏感数据、异常外联、越权文件操作。
5. 性能如何：latency、TTFT、tokens/s、cache hit、token cost。
6. 上游如何：路由目标、健康状态、失败率、fallback 行为。
7. 会话如何演化：多请求 session、失败上下文、长期行为模式。
8. 能否复现：raw cassette 是否足以离线 replay 和构造单元测试。

## 非目标

v1 不以以下事项为核心目标：

- 不做公网多租户“机场”服务。
- 不替代各 provider SDK。
- 不发明一个让用户直接编程的新统一 LLM API。
- 不以 OpenAI/Claude/Gemini 之间的协议互转为主目标。
- 不在默认热路径上自动改写或脱敏用户请求。
- 不让 LLM 总结成为基础审计的前置依赖。

## 产品边界

v1 的产品边界是：

- 透明代理和多上游路由。
- raw 协议录制和 deterministic replay。
- provider-aware 深度协议解析。
- 风险、敏感信息和工具行为审计。
- 可视化调试与指标分析。
- 面向 AI 代理的 MCP 查询与分析接口。

## 产品形态

v1 可以有三种部署形态：

### 本地开发工具

单机运行，服务个人开发和 AI 工具调试。

### 团队自托管服务

部署在团队内网，作为开发/测试环境的 LLM 观测代理。

### 企业审计网关

带访问控制、保留策略、脱敏策略、审计规则和报表导出。

企业形态不是 v1 第一阶段必须完整实现，但架构需要避免未来无法演进。
